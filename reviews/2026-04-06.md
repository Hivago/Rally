## What Was Done

### Bug Fix: GET /api/catalog/restaurants 500 Error
- **Root cause 1:** Npgsql 8.x maps `List<string>` to PostgreSQL `text[]` by default. The `cuisine_types` column is `jsonb`, causing a type mismatch at runtime. Fixed by adding explicit `HasConversion` with `JsonSerializer` + `ValueComparer` in `RestaurantConfiguration.cs`.
- **Root cause 2:** The `AddRestaurantOwnerAndOutletSupport` migration was recorded in `__EFMigrationsHistory` but the columns (`cuisine_types`, `is_pure_veg`, `is_vegan_friendly`, `has_jain_options`, `min_order_amount`, `fssai_number`, `owner_id`) were never actually created in the database. Added idempotent `AddMissingRestaurantColumns` migration using `IF NOT EXISTS` raw SQL.
- **Root cause 3:** `YourMigrationName` migration tried to rename index `ix_restaurants_owner_id` which didn't exist. Made it a no-op.

## Lessons Learned
- **Npgsql 8.x breaking change:** `List<string>` defaults to `text[]`, not `jsonb`. Always add explicit `HasConversion` when using `.HasColumnType("jsonb")`.
- **Migrations can be phantom-applied:** A migration can appear in `__EFMigrationsHistory` without the DDL executing. Always verify columns exist, not just migration records.
- **EF tools base path:** When using `IDesignTimeDbContextFactory`, `Directory.GetCurrentDirectory()` is set by EF to the `--startup-project` path, not the repo root.
- **Always push after commit** — user deploys from remote branch.
