## 2. Where did AI go wrong?

- [x] **Other: Wrong ProjectReference depth** ‚Äî Used `../../../src/...` in both test `.csproj` files when the tests are 4 levels deep (`tests/Modules/Orders/ProjectName/`), so the path should have been `../../../../src/...`. Required a manual fix after the first build attempt failed.

- [x] **Other: Wrong enum value name** ‚Äî Used `CancellationReason.CustomerRequest` instead of the actual enum value `CancellationReason.CustomerRequested`. Should have read the enum file before writing tests.

- [x] **Other: Missing `using Xunit;`** ‚Äî Wrote test files without explicitly importing the Xunit namespace. The project uses `ImplicitUsings` but xunit is not a system namespace, so `[Fact]` was unresolved. Required an extra fix pass.

- [x] **Other: NuGet version conflict** ‚Äî Pinned `Microsoft.Extensions.Logging.Abstractions` to `8.0.0` in the application test project, but the transitive dependency chain (`SharedKernel ‚Üí Extensions.Logging 8.0.1 ‚Üí Abstractions >= 8.0.2`) required `8.0.2`. Caused a restore failure. Should have omitted the explicit reference and let NuGet resolve it transitively.

**Pattern: AI read the domain entities well but skipped reading enums and exact member names before using them in tests. Always read the enum file before writing test cases that reference enum values.**

**What to add to CLAUDE.md to prevent this:**
> When writing tests that reference domain enums, always read the enum source file first to verify exact member names before using them in test code.
